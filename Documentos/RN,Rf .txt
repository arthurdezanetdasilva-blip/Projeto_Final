RF, RNF e Regras de Negócio
Questões conceituais
1. O que é um Requisito Funcional (RF)? Dê a definição e dois exemplos que não estejam no material da aula.
Um Requisito Funcional (RF) descreve uma funcionalidade que o sistema deve executar, ou seja, define o que o sistema deve fazer.
Exemplos:
* RF-01: O sistema deve permitir que o usuário altere sua senha.
* RF-02: O sistema deve emitir um comprovante em PDF após a conclusão de uma compra.


2. O que é um Requisito Não Funcional (RNF)? Explique a diferença entre RNF de desempenho, segurança e usabilidade com um exemplo de cada.
Um Requisito Não Funcional RNF define características de qualidade do sistema, como desempenho, segurança e facilidade de uso.
Desempenho: determina a velocidade e capacidade do sistema.
Exemplo: O sistema deve responder às consultas em até 2 segundos.
Segurança: protege os dados e controla o acesso.
Exemplo: O sistema deve exigir autenticação por senha para acessar informações dos usuários.
Usabilidade: facilita o uso do sistema.
Exemplo: O usuário deve conseguir cadastrar um produto em até 4 cliques.
3. O que é uma Regra de Negócio? Por que ela não é uma funcionalidade do sistema, mas precisa estar documentada?
Uma Regra de Negócio é uma norma ou política da empresa que o sistema deve respeitar. Ela não representa uma funcionalidade, mas determina como determinadas ações devem ocorrer. Deve ser documentada para garantir que o sistema siga corretamente as regras da organização.


4. Por que os requisitos devem ser numerados e escritos no padrão "O sistema deve ? O que acontece quando os requisitos são vagos?
Os requisitos devem ser numerados para facilitar sua organização, rastreamento e manutenção. O padrão "O sistema deve..." torna os requisitos claros e objetivos. Requisitos vagos podem gerar interpretações diferentes, causando erros no desenvolvimento e retrabalho.


5. O que significa validar requisitos com o cliente? Por que isso deve acontecer antes do início do desenvolvimento?
Validar requisitos significa confirmar com o cliente que tudo foi entendido corretamente. Isso evita desenvolver funcionalidades incorretas, reduz custos com mudanças e aumenta a satisfação do cliente.


6. Reflexão
A frase "Quero que o sistema seja seguro e rápido" não é um requisito válido porque é vaga e não pode ser medida. O analista deve conversar com o cliente para definir critérios objetivos, como tempo máximo de resposta e quais mecanismos de segurança serão utilizados.

Questão dissertativa
O requisito "O sistema deve ser seguro e funcionar bem" é problemático porque utiliza termos vagos e subjetivos, que podem ser interpretados de maneiras diferentes. Não é possível medir exatamente o que significa "seguro" ou "funcionar bem", dificultando o desenvolvimento e os testes. Uma forma correta de documentá-lo seria: RNF-01: O sistema deve bloquear o acesso após cinco tentativas consecutivas de login incorretas. RNF-02: O sistema deve carregar qualquer página em até dois segundos para até 500 usuários simultâneos. RF-01: O sistema deve permitir que o usuário altere sua senha quando desejar. Requisitos mal escritos podem gerar dúvidas na equipe de desenvolvimento, aumentar o retrabalho, dificultar a realização dos testes e causar conflitos com o cliente, pois não existe um critério claro para verificar se o sistema atende ao que foi solicitado. Por isso, os requisitos devem ser claros, objetivos, mensuráveis e sem ambiguidades.
